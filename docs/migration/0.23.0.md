# Migrating to 0.23.0

> **Audience.** Operators and AI agents upgrading a consumer project
> from `@bookedsolid/rea@0.22.x` (or earlier) to `0.23.0+`.
> If you are working on the `@bookedsolid/rea` repo itself, see
> [`docs/architecture/bash-scanner.md`](../architecture/bash-scanner.md)
> instead.

## TL;DR

- The Bash-tier scanner that backs `protected-paths-bash-gate.sh` and
  `blocked-paths-bash-gate.sh` is now parser-backed (mvdan-sh AST)
  instead of regex.
- Detection is **strictly stronger** than 0.22.0 тАФ every previously-
  bypassed PoC now BLOCKs.
- New runtime requirement: a working `rea` CLI must resolve through
  one of two sandboxed paths (no env-var override).
- No on-disk verdict format change; consumers needing custom verdict
  handling must switch from stderr-text scraping to JSON parsing.
- `REA_NODE_CLI` env var is silently ignored.
- Roll back to `0.22.0` is non-destructive if a regression surfaces.

## What changed

The 0.23.0 release rewrites the Bash-tier scanner that backs
`protected-paths-bash-gate.sh` and `blocked-paths-bash-gate.sh`.

- The 500+ line bash regex pipeline at `hooks/_lib/cmd-segments.sh` +
  `hooks/_lib/interpreter-scanner.sh` is removed.
- The two hooks are now ~80-line shims that forward stdin to a new
  CLI subcommand `rea hook scan-bash` and verify the verdict JSON
  shape returned.
- Detection lives in `src/hooks/bash-scanner/` тАФ a parser-backed AST
  walker over `BashFile` / `BashNode` from
  [mvdan-sh@0.10.1](https://www.npmjs.com/package/mvdan-sh).

This is a structural refactor that closes 24+ known-bypass classes
across 8 rounds of adversarial review (helix-023 + discord-ops Round
13 + codex rounds 1тАУ8). Detection is strictly stronger than 0.22.0 тАФ
every PoC that bypassed 0.22.0 is now a corpus fixture that BLOCKS.

The other 5 bash-tier hooks (`dependency-audit-gate.sh`,
`env-file-protection.sh`, `secret-scanner.sh` for Bash, etc.) remain
on the regex/segmenter pipeline at `hooks/_lib/`. Migration of those
hooks to the parser-backed scanner is **deferred to 0.24.0**.

## Runtime requirement (NEW)

The shim hooks shell out to the `rea` CLI, which is a Node binary.
Operators MUST have:

- A working `node` on PATH (the shim's verdict-JSON verifier
  requires it).
- A resolved `rea` CLI through one of two sandboxed paths (in order):

  1. `<CLAUDE_PROJECT_DIR>/node_modules/@bookedsolid/rea/dist/cli/index.js`
     тАФ consumer install layout. This is the standard production path.
  2. `<CLAUDE_PROJECT_DIR>/dist/cli/index.js` тАФ rea repo dogfood path
     (only relevant when `@bookedsolid/rea` is the project itself).

The shim verifies that
`realpath(cli).startsWith(realpath(CLAUDE_PROJECT_DIR) + sep)` тАФ
any escape from the project root refuses.

If neither path resolves, the shim refuses the Bash command on
uncertainty (exit 2, stderr `rea: CLI not found тАж`).

`rea doctor` flags missing CLI as a P0 install error.

### Why this is narrower than pre-0.23.0-round-4

Pre-codex-round-4 the resolver had 4 tiers including `rea` on PATH
and `node_modules/.bin/rea`. Both were dropped because:

- PATH lookup is workspace-attacker-controllable (an attacker who can
  drop a binary at `<workspace>/node_modules/.bin/` ahead of `rea`
  on PATH defeats the gate).
- `node_modules/.bin/rea` symlink is workspace-mutable; any package
  install that races the bash gate could overwrite it.

The 2-tier sandboxed resolver is the structural-impossibility claim:
both surviving paths are inside `realpath(CLAUDE_PROJECT_DIR)`, and
the shim refuses any escape via realpath containment.

## Breaking changes

### `REA_NODE_CLI` env var is silently ignored

Codex round 1 F-3 + round 2 R2-3 dropped the env-var path entirely.
Pre-round-2 the shim accepted any `*/dist/cli/index.js` whose verdict
JSON happened to be valid; an attacker who staged
`/tmp/dist/cli/index.js` returning `{"verdict":"allow"}` could
subvert the gate. The env-var path was a security knob without a
real-world use case.

If your test harness or CI relied on `REA_NODE_CLI`, switch to
setting `CLAUDE_PROJECT_DIR` to a directory whose
`dist/cli/index.js` (or `node_modules/@bookedsolid/rea/dist/cli/index.js`)
points at the trusted CLI build. The 2-tier resolver above covers
every legitimate case.

### Verdict-JSON verification is enforced

The shim parses verdict JSON via `node -e` after each invocation and
refuses on any of:

- malformed JSON,
- top-level `verdict` not in `{"allow", "block"}`,
- exit code disagrees with verdict (e.g. exit 0 + verdict "block").

Custom CI wrappers that swallowed the original hook output and
returned 0 will need to allow the verified shape through.

### No bash-pipeline source-of-truth at `hooks/_lib/cmd-segments.sh`

Operators who forked the upstream regex pipeline to add custom
patterns must port their patterns to `src/hooks/bash-scanner/walker.ts`
as a per-utility detector. See the architecture doc for the recipe.

## Audit emissions (NEW)

`rea hook scan-bash` now emits an audit record per invocation under
the tool name `rea.hook.scan-bash` (server `rea`). Fields include:

- `mode`: `"protected"` or `"blocked"`,
- `verdict`: `"allow"` or `"block"`,
- `detected_form`, `hit_pattern` (when block),
- `command_preview` (first 256 chars of the scanned command).

Best-effort тАФ failure to write an audit entry never changes the
verdict.

## Performance

The scanner adds parser overhead to every Bash tool call. Rough
order-of-magnitude (Apple M-series, Node 22):

- Cold start (first call after process spawn): 30тАУ60 ms.
- Warm path (cached parser load): 2тАУ8 ms per typical command.

For interactive Claude Code sessions this is invisible. For CI
pipelines that issue thousands of bash calls in series, factor in
~5 ms per call. The shim spawns `node` once per call; the warm path
is dominated by V8 startup, not parser cost.

## Troubleshooting

Symptoms keyed by the error message you see in stderr or `.rea/audit.jsonl`.

### `rea: CLI not found тАж Refusing the Bash command on uncertainty`

The shim couldn't resolve `rea` through either of the 2 sandboxed
paths.

**Most likely causes:**

1. **`@bookedsolid/rea` is not installed in the project's `node_modules`.**
   Run `pnpm install @bookedsolid/rea` (or `npm install`).
2. **`CLAUDE_PROJECT_DIR` is set wrong.** The shim's containment check
   uses `CLAUDE_PROJECT_DIR` as the project root. If it points at a
   parent or sibling directory, the resolver looks in the wrong
   `node_modules`. Verify with `echo $CLAUDE_PROJECT_DIR`.
3. **Symlink layout escapes the project root.** If
   `node_modules/@bookedsolid/rea` is a symlink to outside the project
   (rare, but happens with hoisted monorepo layouts), the realpath
   containment check refuses. Move the install inside the project or
   use a non-hoisted layout for rea specifically.
4. **`pnpm` workspace hoisting.** Some pnpm workspace shapes hoist
   `@bookedsolid/rea` to a workspace-root `node_modules` outside the
   project subdir. Pin rea to the project subdir's `node_modules`.

**Fix verification:**

```sh
ls -la "$CLAUDE_PROJECT_DIR/node_modules/@bookedsolid/rea/dist/cli/index.js"
# Should be a real file, NOT a symlink to outside CLAUDE_PROJECT_DIR.
```

### `node: command not found` in shim stderr

The shim's verdict-JSON verifier requires Node. Install Node 22+ on
the harness PATH.

### `parse_failure_reason: parser: тАж` BLOCK verdicts

The bash command failed to parse. The scanner fail-closes (BLOCK)
on parse failure because an unparseable command can't be reasoned
about safely.

**Most likely causes:**

1. **Truncated input** тАФ the harness sent a partial command. Check
   the bash pipeline upstream of the hook.
2. **Esoteric bash features mvdan-sh doesn't support** тАФ vanishingly
   rare; if you have a reproducible parse failure, file an issue.
3. **Embedded null bytes** тАФ JSON-tunneled commands sometimes carry
   stray nulls. Strip them upstream.

### `PROTECTED PATH (bash): nested-shell payload is dynamic тАж`

A `bash -c "$VAR"` or `bash -c "$(cmd)"` shape was detected. The
inner payload is computed at runtime тАФ the scanner cannot statically
resolve it, so it refuses on uncertainty.

**Fix:** inline the command instead of wrapping in `bash -c "..."`
with a `$`-expanded payload. If you genuinely need dynamic command
construction (rare), construct the command above the bash gate
boundary and pass the resolved literal in.

### `PROTECTED PATH (bash): xargs destination is fed via stdin тАж`

`xargs CMD` with no static destination тАФ the destination comes from
stdin and the scanner can't see it. Refuse on uncertainty.

**Fix:** rewrite as a `for` loop with explicit destinations, or
pipe to a known-safe directory.

### `PROTECTED PATH (bash): unresolved shell expansion in target.`

A redirect target like `> $VAR` or `> $(cmd)` тАФ dynamic, refuse on
uncertainty.

**Fix:** resolve the variable to a literal path before the redirect.

### Scanner blocks a command you expected to allow

This is over-correction. The scanner's policy is fail-closed.

1. **Check the `detected_form`** in the audit log. It tells you which
   detector fired.
2. **Sanity-check the policy.** Maybe `policy.protected_paths` or
   `policy.blocked_paths` is wider than you intended. Run
   `rea` and check the rendered effective patterns.
3. **If the detector is genuinely wrong** (false positive on a
   common shape), file an issue with the verbatim command. We add
   shapes to the negative corpus regularly.

### Scanner allows a command you expected to block

This is a real bypass. **Do not work around it** тАФ file an issue
immediately with the verbatim command and the protected target. The
codex review pipeline that runs against every release exists to
catch these; if one slipped through, we want to know.

In the interim, consider:

1. Adding the path to `policy.protected_writes` (Bash-tier policy)
   AND an extension-surface allow-list rule if applicable.
2. Setting `.rea/HALT` to lock the harness while the bypass is
   investigated.

## How to verify the install

After upgrading, run:

```sh
# Health check тАФ flags missing CLI, missing node, hook misconfig.
rea doctor

# Run the bash-tier corpus тАФ every documented bypass class should pass.
pnpm vitest run __tests__/hooks/bash-tier-corpus.test.ts

# Optional: run the cross-product corpus (~30s, тЙе7700 fixtures).
pnpm vitest run __tests__/hooks/bash-scanner/adversarial-corpus.test.ts
```

Common manual PoCs you can run yourself to verify the gate is wired:

```sh
# Should BLOCK with PROTECTED PATH (bash):
echo '{"tool_input":{"command":"rm .rea/HALT"}}' | bash hooks/protected-paths-bash-gate.sh
echo '{"tool_input":{"command":"echo \"$(rm .rea/HALT)\""}}' | bash hooks/protected-paths-bash-gate.sh

# Should ALLOW (exit 0):
echo '{"tool_input":{"command":"ls -la"}}' | bash hooks/protected-paths-bash-gate.sh
```

If `rea doctor` flags any "missing CLI" or "shim verdict-verification
mismatch" warning, the hook layer is misconfigured.

## Rollback path

If a regression surfaces in 0.23.0, the rollback is non-destructive:

```sh
# Pin to the last known-good 0.22.x release.
pnpm install @bookedsolid/rea@0.22.0
rea init
```

`rea init` re-installs the bash-pipeline hooks at `hooks/`, which
restores the pre-rewrite behavior. Note that 0.22.0 has the helix-022
fixes but NOT the codex rounds 1тАУ8 fixes тАФ only roll back if you
have a specific blocker on 0.23.0+.

## What's deferred to 0.24.0

- **WASM `sh-syntax` parser evaluation.** The current parser is the
  deprecated `mvdan-sh@0.10.1`. Functionally complete and correct,
  but pinning to an unmaintained library is a known liability. The
  migration path: `parser.ts` is the single touch-point; swapping
  parsers changes one file. Tracked under issue scope "0.24.0
  parser migration."
- **Migrate the other 5 bash hooks to scanner-backed.** Currently
  only `protected-paths-bash-gate.sh` and `blocked-paths-bash-gate.sh`
  use the parser-backed scanner. `dependency-audit-gate.sh`,
  `env-file-protection.sh`, `dangerous-bash-interceptor.sh`, and the
  Bash-tier paths in `secret-scanner.sh` + `architecture-review-gate.sh`
  remain on the regex/segmenter pipeline. Migration is deferred until
  the 0.23.0 scanner stabilizes through 1тАУ2 minor releases.
- **Glob enumeration.** Codex round 1 F-14 conservatively flags
  redirect targets containing glob metachars (`*`, `?`, `[`, `{`)
  as dynamic тЖТ BLOCK. The future enhancement: enumerate glob
  matches against the filesystem at scan-time and check each
  expansion individually. Scoped only to redirect-form because
  argv-based commands legitimately glob (`chmod +x bin/*.sh`).
- **awk `-f script-file` body scan.** Currently we refuse with a
  dynamic detection because the script body is not in our argv. The
  future enhancement: `fs.readFileSync` the script, scan it with
  the awk source patterns. This requires the file to be readable at
  scan-time which adds I/O to the hook critical path; deferred for
  benchmarking.
- **NotebookEdit matcher coverage** for the bash-scanner hooks.
  NotebookEdit is unique to the Claude Code Write-tier surface;
  bash-tier scanner is unaffected, but a parallel rewrite of the
  `settings-protection.sh` matcher coverage is queued.
