### Class G — Cross-file audit-emission contract test

Long-tracked since the 0.16.x audit. helixir flagged across rounds 65/66/73 (rea 0.13.0 → 0.17.0): `commands/codex-review.md` said audit emission was optional, `agents/codex-adversarial.md` said REQUIRED, runtime always emitted. 0.18.0 reconciled the docs to runtime; 0.19.0 ships an automated checker so future drift fails CI.

`__tests__/contracts/audit-emission-contract.test.ts` parses each of the three documents (commands/codex-review.md, agents/codex-adversarial.md, src/hooks/push-gate/index.ts) and asserts:

- Each declares the audit-emission obligation (signal phrases tested).
- Neither markdown document contains the pre-0.18.0 contradiction phrasing (antisignal phrases — fail loudly if they return).
- The runtime `EVT_REVIEWED` constant value is referenced by both markdown documents.
- The runtime's `safeAppend(EVT_REVIEWED)` call lives inside the post-summarize success path (catches future refactors that move the audit emission outside review-completion).

11 tests; pure read-only file parsing; no network; runs in <5 ms.

### Class M — settings.json JSON-schema validation

Long-tracked from the 0.15.0 audit. Catches schema-drift in `.claude/settings.json` matcher group registrations — typos like `"matchers"` (plural), misplaced fields (e.g. `command:` at the matcher level instead of inside `hooks[]`), unknown event names.

`__tests__/contracts/settings-schema.test.ts` validates the document produced by `defaultDesiredHooks()` + `mergeSettings({}, …)` against an Ajv-compiled JSON schema modeling Anthropic Claude Code's documented hook-config shape:

- `hooks.<EventName>` is an array of `{ matcher, hooks: [{type:'command', command, timeout?, statusMessage?}] }`.
- `additionalProperties: false` on the inner `hooks` event-key map so `PreToolUseE` typos surface.
- 8 tests: valid default, group invariants, typo guard, missing-field rejection, unknown-type rejection, timeout floor, every shipped hook command path references `.claude/hooks/`.

New devDependency: `ajv@^8.17.1`.

### npm CDN verify flake (deferred to manual policy approval)

Long-tracked from 0.16.0 release notes: the post-publish `npm pack` verify step in `.github/workflows/release.yml` flakes when the npm CDN propagates the tarball more slowly than the registry record (12 attempts × 10s = 120s budget). Investigated; the right fix is linear-backoff with a 9-min cap (24 attempts, base 10s + 5s/attempt capped at 30s).

The `.github/workflows/release.yml` file is in `policy.blocked_paths`; this release does NOT modify it. Operator can apply the diff manually:

- Replace the 12-attempt × 10s loop in Phase 2 with the 24-attempt linear-backoff loop.
- Cap individual sleep at 30s.

The package IS published when this verify step runs; failure here is a content-drift integrity check, not a release blocker. Manual rerun of the workflow step typically resolves it.
