Fix helix-028 P1 multiline-payload awk bypass in `hooks/_lib/cmd-segments.sh`. Pre-fix, `_rea_unwrap_nested_shells` invoked awk with `-v raw="$cmd" -v masked="$masked"`, which errored with `awk: newline in string ...` whenever the wrapped command contained a literal newline (e.g. `bash -lc "printf x > .rea/HALT\ntrue"` with an embedded newline byte). Awk fell back to the unsplit outer segment, so every gate that depends on the segment-aware helpers (env-file-protection, dependency-audit-gate, attribution-advisory, dangerous-bash-interceptor, security-disclosure-gate, local-review-gate) missed the dangerous first line inside a multiline `bash -c/-lc` wrapper — reopening the nested-shell bypass class `_rea_unwrap_nested_shells` was added to close in 0.17.0.

The fix feeds the entire (possibly multiline) `$cmd` to awk as a single record using a multi-byte record separator (`\x1c\x1d` = FS+GS) so quote-state continuity and newline preservation hold across the whole input. Both the masking pass and the wrapper-scan pass now use NUL-region stdin instead of `awk -v`. Sibling sweep applied the same record-separator fix to `quote_masked_cmd` (which had the identical mode-reset-per-line scrambling bug).

Adjacent fix: ANSI-C dollar-quoted body recognition. Pre-fix `bash -lc $'echo > .rea/HALT\nrm -rf /tmp/foo'` was not extracted because the wrapper-scan only handled `'…'` and `"…"` quoted bodies, and the masker treated `$` as plain text in mode 0 so the closing `'` of `$'…'` flipped the mask state for the rest of the input. Mode 3 (ANSI-C) is now honored in the masking awk, the wrapper-scan awk, the splitter awk, and `quote_masked_cmd`. Common ANSI-C escapes (`\n`, `\t`, `\r`, `\\`, `\'`, `\"`, `\a`, `\b`, `\e`, `\E`, `\f`, `\v`, `\?`, `\xHH`, `\NNN`) are decoded when emitting the payload so the segment splitter sees real newlines and splits on them — every `any_segment_starts_with`-based detector then fires uniformly on wrapped multiline inputs.

Round-1 codex follow-up: the bash helper covers the LEGACY gates that still source `cmd-segments.sh`, but the production protected/blocked-paths bash-tier gate has been a Node-binary scanner since 0.23.0 — and the scanner had the SAME bypass class with broader scope. `walker.ts::wordToString` concatenated `SglQuoted{Dollar:true}` parts verbatim, leaving raw escape sequences (`\n`, `\xHH`, `\NNN`) in the value. Then `protected-scan.ts::stripBashBackslashEscapes` (regex `\\([A-Za-z0-9./_~-])`) mangled `\n` into bare `n`, turning `.rea/HALT\ntrue` into `.rea/HALTntrue` and bypassing the protected-pattern match — even though the real bash runtime would expand `\n` to LF and target `.rea/HALT`. New `decodeAnsiC` helper in `walker.ts` decodes the full bash ANSI-C set (single-char escapes, `\xHH` 1-2 hex, `\NNN` 1-3 octal, `\uHHHH` 1-4 hex, `\UHHHHHHHH` 1-8 hex, `\cX` control char) and is invoked for every `SglQuoted{Dollar:true}` part in `wordToString`. Unsupported escapes mark the word dynamic, fail-closed via the existing dynamic-target block path. Every protected/blocked path detector benefits — redirect target, cp/mv/install/ln dest, nested-shell payload, heredoc body, here-string `<<<`, eval payload, find -exec inner, xargs inner — because they all funnel through `wordToString`.

37 new tests across 3 files: `__tests__/hooks/cmd-segments-helix-028.test.ts` (24 — helper-level), `__tests__/hooks/bash-scanner/walker-ansi-c.test.ts` (13 — scanner unit), `__tests__/hooks/bash-scanner/scan-bash-cli-helix-028.test.ts` (13 — end-to-end CLI integration). The CLI-level tests pin the production gate against every PoC at the same surface a real Claude Code invocation triggers, closing the round-0 test gap (helper-only coverage missed the Node scanner bypass).
