feat(0.49.0): rea init self-pin + bash-gate bootstrap allowlist

Closes the consumer self-brick scenario where `rea init` writes hook
shims that depend on `@bookedsolid/rea` but doesn't add the dep to
`package.json`. Any fresh clone + `pnpm install` produced a repo
where the shims refused 100% of Bash calls — including the very
`pnpm add -D @bookedsolid/rea` that would recover.

Two paired fixes:

1. **`rea init` self-pin.** Adds `@bookedsolid/rea` as a
   `devDependencies` entry with a caret pin against the installing
   CLI's version. Idempotent re-run is byte-identical. Existing
   different version warns and skips (operator owns the pin).
   Dogfood short-circuits when the consumer IS `@bookedsolid/rea`.
   `rea doctor` gains `checkSelfPinDeclared` (FAIL on brick state).
   `rea upgrade` self-heals legacy installs.

2. **Bash-gate bootstrap allowlist.** The `blocked-paths-bash-gate.sh`
   and `protected-paths-bash-gate.sh` shims gain a narrow allowlist
   that passes through `pnpm install`, `npm ci`, `yarn`,
   `corepack enable`, and `pnpm add -D @bookedsolid/rea` (plus
   equivalents) when the CLI is missing AND `package.json` declares
   `@bookedsolid/rea` in `dependencies` or `devDependencies`. Always
   on by default; opt out via `policy.bootstrap_allowlist.enabled:
   false`. Audit event `rea.bash.bootstrap_allow` recorded on every
   match. Multi-segment payloads, non-allowed flags, path-form
   argv, non-rea specs, and forged preconditions all refuse.
   `package.json` is added to `bst-internal*`'s `blocked_paths` to
   close the precondition-forge attack route.

Tampering posture unchanged: round-2 (no env-var escape) and
round-4/5 (no PATH lookup, no symlink, realpath sandbox) all
preserved. The allowlist sits ABOVE the existing CLI-missing refuse
branch, NOT replacing it.

See `THREAT_MODEL.md §5.23` for the full threat model.
